# Fujify preset recreation — status

Goal: recreate Lightroom `.xmp` presets (Camera-Standard base) for the 7 looks the
user lacks (Classic Chrome, Eterna Bleach Bypass, Eterna Cinema, Pro Neg Hi,
Pro Neg Std, Provia, Reala Ace). User already has 4 as `.cube`+`.xmp`
(Astia, Velvia, Classic Neg, Nostalgic Neg).

## What works / what's done
- **Panel extraction** (`extract_panels.py`): the 8 landscape 1920x1007 files in
  `Fujify Luts and XMP/` are 4x3 comparison montages (Camera Standard top-left =
  base; fixed `ORDER`). The 8 portrait 1920x2612 files are single photos, auto-excluded
  via `is_montage()` (periodicity test). 7 unique scenes usable.
- **Montage best-effort cubes** (`build_montage.py` -> `montage_cubes/*.cube`):
  fit a smooth **degree-3 global polynomial** color transform from
  CamStd->preset pixel pairs (aligned per-panel via phase correlation in
  `fit_lut.aligned_pair`). Polynomial > scattered 3D-LUT here: noisy/compressed/
  slightly-misaligned correspondence makes a raw LUT wash out; the polynomial keeps
  contrast. Held-out dE: Classic Chrome ~6 (good), others ~12-15 median (much of that
  is webp/alignment noise in the reference, not fit error — see `_montage_proof.png`,
  visually close).

## Composition — REVISITED, actually works on real images (correcting an earlier call)
Initial LUT-space testing (`derive_T.py`: 3-way T spread mean dE 41, LOO predict
median dE ~14 *on the 33^3 lattice*) looked like a dead end. But that metric weighs
color-space corners real photos never visit. **`validate_velvia_composition.py`**
re-tested on the one look where every form of ground truth exists (Velvia: real
`.cube`+`.xmp`, repo VLog cube, screenshot panels), end-to-end, leave-one-out:
composing Velvia from the *other two* matched looks' derived T and rendering it on
the actual photo scores **screenshot-panel dE med 10.74** vs the **real Velvia
cube's own ceiling of med 9.67** (i.e. within ~1 dE of "as good as it can possibly
get" — real-cube-vs-screenshot noise). Visual proof: `_velvia_composition_proof.png`.
**Conclusion: composition is visually near-indistinguishable from the real cube on
real images — it was never actually broken, just measured wrong.**

`build_composed.py` -> `composed_cubes/` (6 looks, T derived from all 3 matched
looks). Beats the montage poly-fit on every look that has a repo source, several
landing AT or BELOW the 9.7 ceiling:

## Delivery: import .cube directly (xmp codec NOT needed)
Lightroom Classic 7.3+ imports `.cube` files directly as profiles (Develop ▸ Profile
Browser ▸ + ▸ Import Profiles) — it writes Adobe's internal table itself and exposes an
Amount slider, same as the fujify presets. So the proprietary xmp codec below is moot
for the user's use case. See `INSTALL_in_Lightroom.md`. **Recommended deliverables:
`composed_cubes/` (6 looks, composition-derived, best accuracy) +
`montage_cubes/pro_neg_hi_montage.cube` (the 7th, no repo source available).**
`montage_cubes/` (all 7, montage poly-fit) and `repo_cubes_std/` (Panasonic-math
rebase, 6 looks) are kept as fallback/reference sets.

## XMP codec — BLOCKED (proprietary) — only relevant if hand-writing .xmp
`crs:Table_<digest>` is **base-85** (alphabet = printable ASCII 33..126 minus
`" & , ; < > \ _ ~`, ascending; confirmed — only this charset matches) then a
**non-standard / headerless compression** (per-preset lengths differ => compressed;
no zlib/gzip/zstd/lz4/xz/bz2/raw-deflate magic at any offset 0..200). `RGBTable` hex
is a 16-byte MD5 digest (not of the raw blob or ascii blob — likely of the
decompressed table). Not cracked. No off-the-shelf decoder found
(chrisperkles/LightroomPresetsToLUTS approximates from sliders via a HALD CLUT;
doesn't decode the table).
**Delivery path for `.xmp`:** convert the `.cube`s to Adobe **Enhanced Profiles**
(which are XMP and per Adobe docs "can have an RGB->RGB LUT specified using the CUBE
format") via Adobe DNG Profile Editor / Lightroom — Adobe writes the base-85 table
itself. Drive via GUI or computer-use.

## Next: repo look rebased to STD (user-requested plan B)
Repo cubes are accurate full-precision renditions in **V-Log base**. To use on a
Camera-Standard image, apply a **Lumix STD->VLog** transform first, then the repo
cube. Best source for that transform: **matched S9 frames shot in both Standard and
V-Log** (user offered to capture; a colorchart or any varied scene at identical
exposure/WB). Derive T_std2vlog from those pairs (look-independent) ->
`STD_q = repo_VLOG_q(T_std2vlog(x))`. Fallback: published Panasonic V-Log/V-Gamut
math (approximate; Lumix "Standard" curve isn't fully public).
